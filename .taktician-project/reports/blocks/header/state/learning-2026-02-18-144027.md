# Learning Analysis — Revision 9

## Mistake

Revision 8 QA marked keyboard navigation as passing based on code review. However, live browser testing (using taktician_screenshot with keyboard actions) revealed the actual implementations were broken. The selector `[role="menuitem"]` in JS was targeting li elements without tabindex, which silently fail `.focus()` calls in browsers.

## Expected

All keyboard-navigable elements (menu items, tabs, accordions, etc.) should have properly focusable elements and working `.focus()` calls verified in a live browser, not just in code review.

## Root Cause

The Functional QA Agent was following instructions to verify keyboard navigation, but relied on code review instead of live browser testing. The code looked correct in isolation — the li element did have `role=menuitem` — but browsers silently ignore `.focus()` on non-focusable elements. This gap between code appearance and browser behavior is only detectable by actually testing in a browser.

- **Responsible Phase:** functional-qa

## Key Insight

The previous QA cycles verified keyboard navigation by reading JS code and checking that the right functions were called — but never actually tested in a browser. The selector `[role=menuitem]` looked correct in code review because the li elements DO have `role=menuitem`. But li elements without tabindex silently ignore `.focus()` calls. Only live browser testing (using taktician_screenshot with press actions to simulate keyboard input) caught this fundamental disconnect between what code says it does and what browsers actually do.

This is a systemic gap: code review is sufficient for static structure, but interactive behavior requires a browser. The fix is to make live keyboard testing non-negotiable for all keyboard-navigable components going forward.
