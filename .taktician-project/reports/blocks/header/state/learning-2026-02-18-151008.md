# Learning Analysis — Revision 10

## Mistake

Focus-visible styling didn't match hover styling (missing text-accent), outline color override failed due to unlayered global rule, and dropdown min-width didn't account for padding.

## Expected

Focus-visible states should have identical visual treatment as hover states for keyboard accessibility. CSS utilities should override global styles. Dropdown sizing should include padding in width calculations.

## Root Cause

Developer didn't check if a global `*:focus-visible` rule existed or whether it was properly layered. An unlayered rule in `body.css` defeats `@layer utilities` rules regardless of specificity. Additionally, when adding `hover:text-accent`, the developer didn't apply the same styling to `focus-visible` for keyboard users. The three issues reveal a gap in CSS cascade layer awareness and focus-visible/hover state parity checking.

- **Type:** missing-project-pattern
- **Responsible Phase:** development

## Suggestions

### Project-Specific (applied to taktician.md)

1. **[ADD] Accessibility Requirements**: Focus-visible states must match hover state styling. When adding `hover:class` to an element, also add `focus-visible:class` with identical styling. This ensures keyboard users see the same visual feedback as mouse users.
   - *Rationale:* Prevents inconsistent keyboard/mouse feedback and ensures accessibility parity between input methods.

2. **[ADD] Required Patterns**: When adding focus-visible or outline overrides to components, check if a global `*:focus-visible` rule exists in `body.css` or other base stylesheets. If found, ensure it is wrapped in `@layer base` (or the same `@layer` as your override). Unlayered global rules always beat `@layer` rules regardless of specificity. Use `taktician_screenshot` with keyboard actions to verify focus styling works on rendered pages.
   - *Rationale:* CSS cascade layers have strict precedence: unlayered rules > @layer base > @layer components > @layer utilities. An unlayered global rule will silently override Tailwind utility classes unless the rule itself is moved into a @layer.

### General (pending user approval)

1. **[ADD] .taktician/agents/developer-agent.md > Critical Rules**: When implementing focus-visible or focus-ring styles, search the entire theme for global `*:focus-visible`, `*:focus`, or outline rules (especially in `body.css`, `global.css`, or reset stylesheets). If a global rule exists outside a `@layer`, move it into `@layer base`. If you can't modify the global rule, use `!important` or re-declare your style with `@layer base` to ensure it takes precedence. Test with `taktician_screenshot` keyboard actions on the rendered page — never rely on code review alone.
   - *Rationale:* Prevents silent failures where CSS utilities appear correct in code but fail to override unlayered global rules at runtime.

2. **[ADD] .taktician/agents/functional-qa-agent.md > Phase 6: Accessibility Testing**: When testing focus-visible states, always compare them visually to hover states using `taktician_screenshot` with keyboard Tab actions. Verify the focus outline color, text color, and background color match the hover state. Don't skip this check even if code review shows matching classes — CSS precedence issues can cause focus styles to fail at runtime while hover styles work.
   - *Rationale:* Ensures focus-visible styling actually renders as intended, catching CSS layer precedence issues that code review can't detect.
