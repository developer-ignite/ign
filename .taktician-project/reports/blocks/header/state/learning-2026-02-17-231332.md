# Learning Analysis — Revision 8

## Mistake

The Header block's keyboard navigation for dropdown menus had three issues that QA missed:

1. **Focus outline blinking**: The focus outline on the trigger button was animating instead of staying visible and stable.
2. **ArrowDown/Up from trigger not working**: Pressing ArrowDown or ArrowUp from the trigger button didn't move focus into the dropdown items.
3. **ArrowUp/Down inside dropdown not cycling items**: Pressing ArrowUp/Down inside the dropdown menu didn't change which item was selected/focused.

## Expected

- Focus outline should remain visible and stable without animation
- ArrowDown from trigger button should move focus to the first dropdown item
- ArrowUp from trigger button should move focus to the last dropdown item
- ArrowDown/Up inside the dropdown should cycle focus through items
- All three behaviors should work smoothly together as part of standard WAI-ARIA menuitem keyboard interactions

## Root Cause

Three distinct gaps in the keyboard navigation implementation (`blocks/Header/HeaderMain/HeaderMain.js`):

**1. Focus Outline Blinking (Fix 7):**
The button element had `transition-colors` class applied globally, which was animating the `outline-color` property when focus state changed. The outline wasn't removed; it was being animated, making it appear to blink rather than stay stable.

**2. Focus Doesn't Move After Opening Menu (Fix 8):**
The code called `openSubmenu(btn)` to remove the `hidden` class, but immediately tried to focus items with `items[0]?.focus()`. However, the element still had `display:none` or the `hidden` class hadn't finished being processed by the DOM. The browser ignores focus() calls on hidden elements. The fix wraps the focus() in `requestAnimationFrame()` to defer it until after the class removal and reflow complete.

**3. Stale Element References (Fix 9):**
The keyboard handler (`handleSubmenuKeydown`) captured the `items` array when the handler was first attached during initialization (line 351). But if the DOM inside the submenu changed later (content added/removed), this reference became stale. The fix re-queries items on each keydown (line 360-361) to always have fresh references.

- **Type:** missing-context
- **Responsible Phase:** development

## Root Cause Analysis

These are not design issues or ambiguous spec gaps—they're implementation oversights that stem from incomplete understanding of:

1. **CSS transitions affecting ARIA focus states**: When interactive elements have `transition-*` classes, developers need to ensure outline/ring properties are explicitly set to non-transitioning values in the base state to prevent animations during focus changes.

2. **DOM visibility and focus sequencing**: Removing a `hidden` class or changing `display` requires a layout recalculation. Focus commands issued before that recalculation completes will fail silently. This is a timing issue, not a logic issue.

3. **Dynamic DOM and event handler caches**: JavaScript event handlers are often written once and attached once. But if the target elements can change between attachment and handler execution, capturing references at initialization time is unsafe. Re-querying on each event is safer for dynamic contexts.

These gaps are specific to JavaScript's DOM APIs and timing—not something that would naturally be caught by reviewing block structure or color tokens. They require hands-on interactive testing with keyboard input and focus tracking.

## Suggestions

### General (pending user approval)

1. **[ADD] .taktician/agents/developer-agent.md > Interactive Elements & Focus Management**:
   - *Content:* When implementing keyboard navigation with dynamic DOM changes (opening/closing menus, showing/hiding elements): (1) Use `requestAnimationFrame()` after removing `hidden`/`display:none` classes to defer focus operations until the element is actually rendered in the DOM; (2) Re-query DOM elements inside event handlers rather than capturing references at initialization, since the DOM tree may change between when the handler is attached and when the event fires.
   - *Rationale:* Focus management requires elements to be visible in the rendered DOM; transitions and class removals need time to complete before focus can move. Stale DOM references cause keyboard navigation to fail silently.

2. **[ADD] .taktician/agents/functional-qa-agent.md > Keyboard Navigation Testing**:
   - *Content:* When testing keyboard navigation in interactive components (dropdowns, menus, tabs): verify all three aspects: (1) focus outline visibility and stability (not animating or blinking), (2) focus movement from trigger/parent to child items, (3) focus cycling within lists. Test both opening the menu AND navigating to items in a single test sequence. Use `taktician_screenshot` with keyboard actions (e.g., `press('ArrowDown')`, `waitForTimeout`) to verify focus changes visually.
   - *Rationale:* Keyboard navigation has multiple failure modes that won't all be caught by visual inspection alone—focus outline animations, stale element references, and focus trap issues require interactive testing.

3. **[ADD] .taktician/agents/planning-agent.md > Phase 9: Generate QA Plan**:
   - *Content:* For blocks with keyboard-driven interactions (dropdowns, modals, carousels with keyboard support), add custom QA tests to the qa-plan that specify the exact keyboard sequence to test. Include checks for: (1) focus outline stability across state changes, (2) focus movement from parent to child elements, (3) focus cycling through lists, (4) escape key closing interaction. Reference WAI-ARIA keyboard interaction specs if applicable (e.g., menuitem role requires ArrowUp/Down to cycle items).
   - *Rationale:* Standard QA rules cover structure and visual design but miss keyboard interaction subtleties. Explicit qa-plan entries ensure QA agents run comprehensive keyboard tests, catching focus management bugs before release.

### Summary of Suggestions

These three suggestions address the workflow and testing gaps that allowed keyboard navigation bugs to slip through:

- **Developer Agent** needs clearer guidance on timing and caching in dynamic DOM contexts.
- **Functional QA Agent** needs explicit instruction to test keyboard navigation interactively, not just visually.
- **Planning Agent** needs to require explicit keyboard test sequences in the qa-plan for interactive components.

All three changes are **general** (apply across all projects, not just IGN) because keyboard navigation and DOM timing are universal JavaScript challenges.
