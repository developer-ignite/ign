# Learning Analysis — Revision 2

## Mistake

Logo only fills the first row (eyebrow/utility row) of the header instead of spanning both the first and second rows.

## Expected

Logo should stretch vertically across both the eyebrow/utility row AND the main navigation row, filling the full height of the header's right column.

## Root Cause

The Planning Agent interpreted Figma's `items-center` property name as indicating vertical centering alignment, when the actual Figma structure required analyzing the component hierarchy. In Figma's layout:

- Logo and `<nav>` were siblings at the flex-row level
- The row contained both a utility section and a navigation section
- When logo and nav are flex siblings at the same level, the logo's `self-stretch` can only extend to the height of the shortest sibling (the utility row)

The fix required moving `<nav>` to be inside a column element alongside the utility row, making the right column taller and allowing the logo to stretch to fill it. This is a structural HTML change, not a CSS alignment adjustment.

The Planning Agent focused on CSS property names rather than verifying that Figma's component nesting supported the intended visual result.

- **Type:** design-interpretation
- **Responsible Phase:** planning

## Suggestions

### Project-Specific (applied to taktician.md)

1. **[ADD] Component Preferences**:
   - *Content:* When rebuilding header/navigation structures: verify that layout elements allowing flexible height (logo, nav bars) are siblings of height-containing elements (columns with multiple rows), not siblings of individual rows. Use `items-stretch` on flex parents to ensure all flex children match the parent's full height, especially when combining elements of different natural heights.
   - *Rationale:* Header logo-spanning-rows issue revealed that flex siblings have height constraints based on the shortest sibling. Planning Agent must verify component nesting supports the intended visual spanning.

### General (pending user approval)

1. **[ADD] .taktician/agents/planning-agent.md > Phase 2: Analyze Figma Design**:
   - *Content:* When analyzing flex layouts in Figma, prioritize component hierarchy (parent/child nesting) over individual property names. Verify that elements intended to span multiple related items are siblings of a container holding those items, not siblings of the items themselves. For example, a logo spanning two rows should be a flex sibling of a column containing both rows, not a flex sibling of individual rows.
   - *Rationale:* Planning Agent misinterpreted items-center as a layout constraint instead of analyzing whether the flex structure allowed multi-row spanning, preventing correct hierarchy early.

2. **[ADD] .taktician/agents/design-agent.md > Mode 3: Quality Assurance**:
   - *Content:* When validating layout relationships in QA, verify not just individual element properties but also structural span patterns: does element X visually span elements Y and Z? Compare the component's nesting hierarchy against the Figma structure to ensure container relationships (which elements are siblings vs. children) match the design.
   - *Rationale:* Design QA validated revision 1 despite logo not spanning both rows because it checked individual properties (alignment, color) but not structural layout relationships.
