# Learning Analysis — Revision 1

## Mistake

Planning agent specified `btn-tertiary` for the external link button without user guidance. Also added an optional `useTopicColor` attribute (topic color override toggle) that was never requested. Both were implemented correctly per the spec but did not match user expectations.

## Expected

External link button should use `btn-primary` style. No topic color override attribute — only standard page accent color for the gradient background.

## Root Cause

Planning agent made design decisions (button styling and optional color override feature) without consulting the user or having them specified in requirements. No source document (spec, Figma, or user guidance) requested these design choices. The agent invented features without asking for clarification.

- **Type:** missing-general-rule
- **Responsible Phase:** planning

## Suggestions

### General (pending user approval)

1. **[ADD] .taktician/agents/planning-agent.md > Phase 2: Analyze Figma Design**: **Decision Rule: Button Styling Without Sources.** Never invent button style choices (btn-primary, btn-secondary, btn-tertiary) without explicit source (Figma, spec, pattern reference, or Design Agent recommendation). If no source exists, ask the user. Tertiary buttons are lowest-priority actions; primary/secondary are higher priority. Choosing button style requires understanding the interaction's importance, which only the user/designer can determine.
   - *Rationale:* Button style communicates visual hierarchy and action importance. Guessing a button style leads to rework. Requiring a source forces agents to ask clarifying questions earlier.

2. **[ADD] .taktician/agents/planning-agent.md > Phase 2: Analyze Figma Design**: **Decision Rule: Optional Features Without Requirements.** Never add optional toggles/attributes (e.g., `useTopicColor`, `useCustomTheme`) unless explicitly requested in spec or Figma or justified by a clear use case documented in the spec. Optional attributes should serve a real authoring need — color overrides need a reason. If unsure, ask the user.
   - *Rationale:* Adding optional features increases complexity and requires user testing. Features without clear authoring intent often get removed in QA/revision. Requiring justification prevents feature creep.
