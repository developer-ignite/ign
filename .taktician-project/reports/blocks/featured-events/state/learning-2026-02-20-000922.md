# Learning Analysis — Revision 2

## Mistake

Six interconnected issues across typography, layout, and styling revealed a systematic chain of literal interpretation errors from structural analysis through planning to development:

1. **Heading too small** — headingSize:3 mapped to 32px/40px, but Figma shows 56px. Planning agent mapped H3 style name without verifying pixel values.
2. **Uppercase class on card title** — Developer added `uppercase` class assuming Anton font inherently needs stylization.
3. **Badge absolute positioning** — Used `absolute` instead of flex flow. Literal frame extraction without analyzing layout intent.
4. **Tertiary button black on hover in dark** — Dark class removed in rev1 but button hover state depends on dark context for correct color.
5. **Fixed height md:h-[349px]** — Frame dimension treated as CSS constraint. Structural analysis → planning spec → developer implementation all accepted literally.
6. **Image missing aspect ratio** — Used fixed height instead of aspect ratio constraint. Content-resilience check failed across planning phase.

## Expected

- Heading size matches Figma visually (56px, not H3 mapping)
- Card title uses natural Anton font without artificial uppercase styling
- Badge positioned via flexbox following Figma layout flow, not frame coordinates
- Tertiary button hover color correct in dark mode context
- Image uses `aspect-ratio` for responsive scaling, not fixed height
- All positioning and sizing extracted with semantic intent, not literal frame coordinates

## Root Cause

Multi-phase systemic issue rooted in mechanical interpretation at each stage:

1. **Planning phase:** Mapped style names (H3) to headingSize attributes without pixel-level verification. No validation that visual size matches mapped token.
2. **Structural analysis phase:** Extracted Figma frame dimensions as literal layout values. Treated coordinates and sizes as CSS truth instead of analyzing layout intent.
3. **Development phase:** Implemented specification mechanically without questioning whether frame dimensions should become CSS constraints.
4. **Missing validations:** No content-resilience check for images (should use aspect-ratio, not fixed height). No dark-mode context auditing when styling dark-mode-dependent components.
5. **No semantic analysis:** Each phase accepted literal values without asking "is this layout intent or frame metadata?"

User specifically asked "who decided md:h-[349px]?" — highlighting that the error cascaded: structural analysis extracted it → planning spec included it → developer implemented it → nobody questioned whether a frame dimension should become a CSS height constraint.

- **Type:** Missing general rule (workflow-level validation gate)
- **Responsible Phase:** Planning (first point where validation should occur)

## Suggestions

### Project-Specific (applied to taktician.md)

1. **[ADD] Typography Rules**: When mapping font sizes from Figma style names (H1, H2, H3) to Tailwind/theme tokens, ALWAYS verify the pixel size matches the intended visual scale in Figma. Style names are guides, not truth. Check Figma design context output for exact px values and cross-reference with headingSize attribute documentation.
   - *Rationale:* Prevents planning agents from mapping style names to wrong size attributes without verification.

2. **[ADD] Typography Rules**: Anton font is not italic and does not have built-in styling. Never add `uppercase` or `italic` classes to Anton-based text unless explicitly shown in Figma. The font's natural slant is a design characteristic, not a CSS transform.
   - *Rationale:* Clarifies that Anton font styling decisions require explicit Figma evidence, not assumptions based on visual appearance.

3. **[ADD] Layout Patterns**: When implementing badge or status elements positioned at corners of cards, use flex/absolute positioning per Figma layout flow (not literal Figma frame dimensions). If Figma shows an element in a corner via flexbox or flow layout, extract the positioning intent, not the frame pixel coordinates.
   - *Rationale:* Prevents literal extraction of frame coordinates which don't represent actual CSS layout intent.

4. **[ADD] Component Preferences**: Tertiary button styling (`btn-tertiary`) depends on dark-mode context for correct hover states. When dark mode is applied to parent section via `dark` class, ensure button styling inherits the dark context. Never remove parent `dark` class without auditing child button hover/focus states.
   - *Rationale:* Documents that tertiary buttons require context awareness for color consistency in dark mode.
