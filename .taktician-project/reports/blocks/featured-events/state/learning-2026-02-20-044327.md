# Learning Analysis — Revision 1

## Mistake

Six major visual and functional issues passed both Functional QA and Design QA:

1. Section-wide dark background was applied to full section instead of only the card halves
2. Date badge positioned directly against the image frame with no spacing
3. Card backgrounds missing the dark-pattern texture overlay
4. Tertiary button missing the required arrow icon
5. Day number (40px) and title (48px) font sizes were incorrect
6. Location/venue field not rendering because test events had no venue data

## Expected

**Design QA expectation:** All 5 visual differences should have been caught by pixel-comparing implementation screenshots against Figma during Step 3 Detailed Visual Comparison. Visual comparison (screenshots) is the primary evidence, not code review.

**Functional QA expectation:** The missing location rendering should have been caught by testing with complete event data that includes all optional fields (venue, location, description, images). Content completeness is a functional requirement.

## Root Cause

Both QA phases relied on verification approaches that bypassed visual and content validation:

- **Design QA** skipped visual screenshot comparison and substituted code review (reading CSS classes, checking token values, reviewing conditional logic). Code review catches obvious mistakes but cannot detect 5px spacing differences, missing texture overlays, or button icon presence/absence.

- **Functional QA** tested with incomplete test event data (events lacking venue field). The QA scripts verified that the conditional rendering of the location output worked in code, but never tested with actual venue data to confirm the feature renders on the frontend.

The user's question "how did all those visual differences passed qa?" reveals the core issue: visual QA did not include visual verification.

- **Type:** missing-context
- **Responsible Phase:** qa

## Analysis Notes

The core lesson is that **QA requires evidence-based verification, not process-based verification.**

- **Design QA** had the right tools (taktician_screenshot, taktician_page_audit) but did not use them for visual comparison. The instructions state "You MUST visually compare screenshots before checking any code" but the Design QA agent interpreted this as optional when code review appeared sufficient. The result: 5 visual issues that were trivial to spot in a screenshot (missing spacing, wrong font sizes, missing texture, wrong button styling) all passed.

- **Functional QA** had the right approach for testing rendering but incomplete test data. The agent verified the conditional logic worked but never tested with complete data. The result: a missing field that would render nothing on the frontend passed as "working as specified."

Both issues stem from agents using **sufficient** verification (code is correct) instead of **complete** verification (visual + functional output is correct). The fix is to make screenshot review mandatory (observable proof) and content completeness mandatory (all paths tested).
