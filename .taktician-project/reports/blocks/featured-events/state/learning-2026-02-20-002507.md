# Learning Analysis — Revision 3

## Mistake

Two layout/spacing issues passed QA and were reported by the user as broken:

1. **Image aspect ratio on desktop**: Images were rendering as thin vertical strips instead of maintaining the 4:3 proportion shown in Figma. Developer attempted fix with `md:aspect-auto md:h-full` which had no effect.

2. **Excerpt spacing before CTA button**: Text group was bunched directly against the CTA button with no spacing. Developer attempted fix with `justify-between` on the content div, which had no effect.

Both issues were visually observable on the rendered page compared to Figma design.

## Expected

- Images maintain 4:3 aspect ratio at all breakpoints (desktop, mobile, tablet)
- Excerpt text has 24px margin-bottom spacing before the CTA button, matching Figma spec
- Visual proportions and spacing match the design system when viewed side-by-side with Figma

## Root Cause

The QA agent failed to perform visual comparison of rendered output against Figma design. Both issues are immediately observable when:

1. **Image aspect ratio**: Take a screenshot and visually measure image proportions (rendered width:height ratio). Compare to Figma design's 4:3 ratio. A distorted aspect ratio is visible as a thin vertical strip vs. the wider proportional image in Figma.

2. **Spacing issue**: Take a screenshot and visually measure the gap between the text group and button. Compare to Figma spacing (24px). A missing gap is visible as text touching or nearly touching the button in the implementation vs. spaced apart in Figma.

The developer's attempted fixes reveal incomplete CSS understanding:
- `md:aspect-auto md:h-full` on a child element inside a constrained parent is ineffective
- `justify-between` on a `flex-col` layout cannot create vertical spacing (justify-between applies to the flex axis, which is vertical in flex-col — the property spaces items at start/end, not between them on the main axis)

However, the systemic failure is **QA not catching visual mismatches**. The issues passed QA because pixel-level screenshot comparison to Figma was not performed.

- **Type:** Design interpretation (QA method failure)
- **Responsible Phase:** Functional QA / Design QA

## Suggestions

### General (pending user approval)

1. **[ADD] .taktician/agents/design-agent.md > Mode 3: Quality Assurance — Step 3.5: Explicit Pixel Comparison (MANDATORY)**:
   - When verifying image aspect ratios in multi-column layouts (e.g., image panels, card galleries), take screenshots at every breakpoint AND manually measure rendered aspect ratios (width:height visual proportions). Compare to Figma measurements. This catches constraint failures where the image container has conflicting width/height/aspect-ratio rules that distort proportions visually despite correct CSS utilities appearing in code. Use browser DevTools to measure bounding box dimensions if unsure.
   - *Rationale:* Image aspect ratio issues are frequently caused by conflicting parent/child constraints that look correct in code but visually distort the element. Code review alone misses these failures.

2. **[ADD] .taktician/agents/design-agent.md > Mode 3: Quality Assurance — Step 3: Detailed Visual Comparison**:
   - For card/panel layouts with multiple text elements (heading, subtext, excerpt, CTA), verify spacing between each element by visual measurement, not by checking for flex/justify utility classes. Spacing can fail due to applied utilities being incompatible with the container layout direction (e.g., justify-between on flex-col). Measure vertical gaps from screenshot and compare to Figma spec for each element pair.
   - *Rationale:* Flex utilities can appear correct in code but fail to apply if the layout direction doesn't match their axis. Visual measurement catches these failures immediately.
