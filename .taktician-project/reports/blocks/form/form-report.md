## Block Behavior

### Layout Modes

The Form block supports two layout modes controlled by the `headingPosition` attribute:

**Beside Layout (default)**
- Two-column grid on desktop (768px and above)
- Left column contains heading content (sticky at scroll)
- Right column contains form
- Stacks to single column on mobile (full-width)
- Sticky heading stays visible during form scrolling on desktop

**Above Layout**
- Centered, stacked design at all breakpoints
- Content and form stack vertically
- Maximum width constraint applied for readability
- Content remains centered on screen

### Content Sections

**Heading Area** (sticky in beside mode)
- Eyebrow text appears above main heading (hidden when empty)
- Main heading serves as section identifier for block naming
- Description text appears below heading (hidden when empty)
- Buttons appear below description (hidden when empty)

**Form Area**
- Displays form via Gravity Forms shortcode when formId is provided
- Completely hidden when formId is empty
- Handles missing Gravity Forms plugin gracefully with no error display

### Conditional Behaviors

- Eyebrow hidden automatically when left empty via shouldDisplay()
- Description hidden automatically when left empty via shouldDisplay()
- Buttons hidden automatically when array is empty via shouldDisplay()
- Form section omitted entirely when formId is empty or Gravity Forms plugin is inactive
- Block title in editor reflects the heading text

### Interactive States

- Form elements inherit Gravity Forms styling with focus indicators defined in global forms.css
- All form inputs (text, select, textarea, checkbox, radio) have visible focus indicators
- Focus indicators use the theme's accent color with proper contrast
- Submit buttons show clear focus and hover states

## Development Notes

### Accessibility Decisions

**Landmark Pattern Implementation**
The block follows WAI-ARIA landmark patterns by using the `theme_block_props()` helper, which automatically adds an `aria-labelledby` attribute on the outer `<section>` element when a heading attribute exists. This makes the section a proper named "region" landmark that assistive technology users can navigate to directly. The heading ID is automatically generated by `theme_block_region_id()`:
- In PHP: `theme_block_region_id()` generates unique IDs with pattern `auto-{hash}-heading`
- The helper automatically adds `aria-labelledby` pointing to this ID
- No manual aria-labelledby prop needed; theme helpers handle this fully

**Form Landmark Handling**
The block does not add an explicit `role="form"` to the wrapper div because Gravity Forms already renders its own `<form>` element inside. Adding a duplicate form landmark would create accessibility issues. Instead, the outer section with aria-labelledby serves as the named region, and Gravity Forms' own form element provides the form landmark.

**Focus Visibility**
Global forms.css provides comprehensive focus-visible styles for all Gravity Forms interactive elements to ensure keyboard users can see which element has focus. Styles use the theme's accent color with box-shadow to meet WCAG contrast requirements.

### Layout Decisions

**Gap Value Correction**
The block uses `gap-y-16` throughout (both TSX and PHP) to match the block-type documentation specification. This was corrected from an earlier `gap-y-10` value to ensure consistency with design requirements.

**Sticky Heading Implementation**
In the beside layout, the heading column uses `md:sticky md:top-[calc(var(--fixed-elements-height)+24px)] self-start` to stay visible during form scrolling. The offset accounts for fixed header elements (via CSS custom property) plus 24px padding.

**Automated Accessibility ID Handling**
The block relies entirely on theme helpers for accessibility ID management. The `theme_block_props()` helper automatically adds `aria-labelledby` when a block has a heading, and `ThemeHeading` defaults to `theme_block_region_id()` for generating the heading ID. This eliminates manual ID prop handling and ensures consistency across all blocks.

### Graceful Degradation

The block includes a check before rendering the form output:
```php
function_exists('gravity_form') || shortcode_exists('gravityform')
```

If the Gravity Forms plugin is deactivated, the form div is not rendered at all, preventing broken shortcode syntax from appearing on the frontend. The heading content still displays normally.

### Color Mapping

Global forms.css uses the following theme tokens:
- Accent color (`--accent-color`) for focus-visible states on all form elements
- Text color for labels and form text
- Background color for input/select/textarea backgrounds
- Error color for validation error states

These styles are comprehensive and properly maintained in the global stylesheet.
