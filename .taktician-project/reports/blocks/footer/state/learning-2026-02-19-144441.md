# Learning Analysis — Revision 3

## Mistake

FooterInfoSocialItem.tsx used a heavy component tree — `__experimentalLinkControl + BlockControls + Popover` for link editing, plus `MediaUpload` inside `IconPicker` for logo selection. This triggered core-data store operations (`editEntityRecord`) during the block insertion lifecycle, before the entity record was ready. The 'Cannot read properties of undefined (reading meta)' console error appeared when adding new social media items, specifically during the WordPress core-data synchronization phase.

## Expected

Adding a new social media item should work without console errors on the first click. The block should be ready to accept link and icon configuration immediately after insertion.

## Root Cause

The heavy component tree forced WordPress core-data to run `editEntityRecord` during block insertion, before the entity meta fields were initialized. The race condition occurred specifically in FooterInfoSocialItem's lifecycle, not in the parent Footer.tsx (which had been fixed in Revision 2 with a narrowed useEffect dependency). Replacing the heavy components with a lightweight `Link` component from `@taktdev/components` (which doesn't trigger core-data ops) and deferring `IconPicker` rendering via an `isSelected` conditional until user interaction eliminated these premature store operations.

- **Type:** missing-project-pattern
- **Responsible Phase:** development

## Solution Summary

**CHANGE-001:** Replaced `__experimentalLinkControl` + `BlockControls` + `Popover` + `ToggleControl` + `TextControl` with the `@taktdev/components Link` component. Link wraps LinkControl internally and handles the focus-triggered popover, open-in-new-tab toggle, and aria-label field without requiring separate editor UI components or BlockControls.

**CHANGE-002:** Added `isSelected` prop to EditProps. When `isSelected=true`, the IconPicker (with its MediaUpload) mounts and renders. When `isSelected=false`, a lightweight static fallback renders — either the matched SVG from `suggestedIcons` or a placeholder span. This prevents MediaUploadCheck + MediaUpload from mounting during block insertion.

**CHANGE-003:** Added explicit defaults to block.json: `logoId: null` and `iconSlug: ""`. Prevents undefined attribute values during block creation.

## Suggestions

### Project-Specific (applied to taktician.md)

1. **[ADD] Component Preferences**: Defer heavy Gutenberg components (MediaUpload, __experimentalLinkControl, Popover with form controls) until block is selected (isSelected=true). Render lightweight static fallbacks during insertion/unselected state. This prevents core-data store operations from firing during block creation lifecycle, which can race with entity meta initialization and cause 'undefined' errors.
   - *Rationale:* Heavy components trigger editEntityRecord during insertion before entity is ready. Deferring to selected state allows entity registration to complete before store operations begin.

2. **[ADD] Component Preferences**: For link editing in child blocks, prefer @taktdev/components Link component over @wordpress/block-editor __experimentalLinkControl. Link wraps LinkControl internally without requiring separate BlockControls, Popover, ToggleControl, or TextControl imports, reducing component tree size and eliminating unnecessary store bindings during insertion.
   - *Rationale:* Lightweight wrapper components reduce surface area for accidental store subscriptions and are safer during block lifecycle events. Link component encapsulates link UI patterns while reducing imports and cognitive overhead.
