# Learning Analysis — Revision 2

## Mistake

Footer.tsx was subscribing to the entire nested block tree via `getBlocks(clientId)`, causing `useEffect` to fire on any deeply-nested block change (e.g., adding a FooterInfoSocialItem, editing any inner block's attributes). This triggered `replaceInnerBlocks` mid-insertion, racing against WordPress core-data entity registration and producing 'Cannot read properties of undefined (reading meta)' console errors. The error only occurred on first click due to timing; second click succeeded because the entity was by then registered.

## Expected

Footer.tsx should only track direct children (template structure) and ignore deeply-nested changes. First click should work without console errors.

## Root Cause

The `useEffect` dependency was on `innerBlocks` object (from `getBlocks`), which changed whenever ANY block inside the tree was modified or inserted. The effect ran `getBlocks` again, computed a new reordered array from the updated tree, called `replaceInnerBlocks`, and this raced with core-data's `editEntityRecord` for newly-inserted blocks. The Planning Agent identified this as a race condition but the Developer Agent's initial fix attempt was incomplete — it only changed the dependency array comment without implementing the actual narrowed selector.

- **Type:** missing-general-rule
- **Responsible Phase:** development

## Suggestions

### Project-Specific (applied to taktician.md)

1. **[ADD] Component Preferences**: For blocks with template enforcement via useEffect + replaceInnerBlocks (e.g., Footer, Header, PageLayout): compute a narrow dependency key from direct children only (string of 'name:clientId' pairs). Never subscribe to getBlocks().find() or any nested property access. This prevents race conditions when inner blocks are inserted or modified deeply within the tree.
   - *Rationale:* Narrow direct-children tracking prevents race conditions between replaceInnerBlocks and core-data entity registration during nested block insertion.

### General (pending user approval)

1. **[ADD] .taktician/agents/developer-agent.md > Phase 2: Create Block Files**: For blocks with inner blocks and template enforcement via useEffect + replaceInnerBlocks: ALWAYS narrow the selector to track only direct children structure (e.g., a string key of direct children's names and clientIds), not the entire nested tree. This prevents the effect from firing on deeply-nested insertions, which can race with WordPress core-data entity registration. Use eslint-disable-next-line react-hooks/exhaustive-deps when a variable from the selector is intentionally omitted from deps (with comment explaining why it's always in sync with the actual dependency).
   - *Rationale:* Deeply-nested block changes in templates cause race conditions with core-data. Narrowing the selector scope prevents mid-insertion replaceInnerBlocks calls that conflict with entity registration.
