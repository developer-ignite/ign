# Learning Analysis — Revision 1

## Mistake

Developer agent instructions lack explicit tool selection guidance, creating ambiguity about when to use Edit vs Write for file modifications, or when to use taktician tools vs Bash for WordPress operations.

## Expected

Clear, enforced rules in developer-agent.md that guide agents to:
1. Use Edit for modifying existing files (default behavior)
2. Use Write only for creating new files
3. Use taktician tools before Bash for WordPress/theme operations

## Root Cause

The developer-agent.md instructions specify WHICH tools to use in specific phases (Phase 6 for test pages uses `taktician_wp_update`, Phase 5.5 for validation uses `taktician_validate_all`), but do not establish a general principle about tool selection. Agents must infer correct tool usage from generic tool documentation rather than from explicit block-level developer guidance. This creates risk that future agents will default to less optimal tool choices.

- **Type:** ambiguous-instruction
- **Responsible Phase:** development

## Suggestions

### General (pending user approval)

1. **[ADD] .taktician/agents/developer-agent.md > Tool Restrictions**: Add explicit rule that Edit is the default tool for modifying existing block files (.tsx, .php, .json, .css, .svg). Write is ONLY for creating new files. Bash is ONLY for operations that cannot be performed with other tools.
   - *Rationale:* Establishes clear precedence to prevent agents from using destructive Write operations on existing files, which can lose work if the file isn't read first.

2. **[ADD] .taktician/agents/developer-agent.md > Tool Restrictions**: Add explicit rule to always use `taktician_wp_*` tools for WordPress operations (list, get, create, update, delete posts/pages/media) before falling back to Bash. Always use `taktician_wp_raw_content` or `taktician_wp_count_blocks` to verify block instances exist on test pages, never use `taktician_wp_get` (which returns rendered HTML with stripped block comments).
   - *Rationale:* Ensures consistency across agents and prevents silent failures from Bash commands that don't validate WordPress state properly.
