### Analysis

**Mistake 1 - Black text on black background (pagination in dark mode)**

- **What was wrong:** The `.page-numbers.current` element in `resources/css/screen/navigation.css` used `bg-black text-white` for its default (light mode) state but had no dark mode overrides. When rendered inside a `dark` section (the Archive block uses the charcoal dark mode pattern), the inherited dark mode text color reverted to dark/black while the background remained black, producing invisible text.
- **What was expected:** Pagination text should remain readable in dark mode sections, with inverted colors (white background, charcoal text).
- **Root cause:** The pagination component is a shared/global CSS component (`navigation.css`), not part of the block itself. The Developer Agent built the block's own elements correctly for dark mode but did not audit shared components that would render inside the dark section. The pagination markup is generated by WordPress core (`paginate_links()`), not by the block template directly, so the Developer Agent had no reason to look at `navigation.css`.

**Root cause type:** Missing general rule

**Responsible phase:** QA (both Functional QA and Design QA)

**Why QA missed it:**

1. **Functional QA** checked pagination structure (aria-labels, link interception, nav element) but does not verify visual rendering or computed CSS styles. Its instructions include a "Color Contrast" check under accessibility (Phase 5) that says "Text meets WCAG AA: 4.5:1 for normal text, 3:1 for large text" -- but the `taktician_page_audit` tool only collects DOM structure (headings, child counts, data attributes, ARIA attributes, images, links, selectors). It cannot report computed `color` and `background-color` values, so this check is structurally impossible with current tooling.

2. **Design QA** compared implementation screenshots against Figma designs. But the Figma designs only showed a "Load More" button for pagination -- there was no Figma reference for standard numbered pagination. Without a Figma reference, Design QA had nothing to compare the pagination against, so it was never visually inspected for correctness. Design QA noted "Dark theme context (text-white on dark background) correctly applied" as a positive finding, but this observation was about the block's own heading/description elements, not about shared global components rendering within the dark section.

3. **The systemic gap:** Neither agent checks whether shared/global CSS components (pagination, form elements, WordPress core output) maintain correct contrast when rendered inside `dark` mode sections. The block itself was correct -- the problem was in CSS that existed before the block was created.

---

**Mistake 2 - Missing spacing between heading, filters, and grid**

- **What was wrong:** No vertical gap between the heading area, filter form, and card grid inside the dark mode wrapper.
- **What was expected:** Visible vertical separation between these major sections.
- **Root cause type:** Missing project pattern
- **Responsible phase:** Development (spec may have lacked explicit gap values for the dark mode wrapper and archive-inner container)

---

**Mistake 3 - Block icons not following guidelines**

- **What was wrong:** The original SVG icons used strokes, rounded corners (`rx`/`ry`), circles, and `fill="white"` -- all explicitly prohibited by the block icon guidelines.
- **What was expected:** Icons following the documented rules: filled paths only (no strokes), integer coordinates, no rounded corners, `currentColor` fill only.
- **Root cause type:** Ignored instruction
- **Responsible phase:** Development (Developer Agent created icons without following the block-icons.md reference)
