# Learning Analysis — Revision 1

## Mistake

The developer created a separate `relatedNonSingularBehavior` attribute to handle fallback behavior when displaying related posts on non-singular pages, rather than using the generic `fallbackToLatest` attribute shared across all modes.

## Expected

A single `fallbackToLatest` attribute shared across automatic and related modes that works on both singular and non-singular pages, combined with `hideIfEmpty` to control empty-state rendering.

## Root Cause

The developer interpreted "fallback on non-singular pages" as requiring a separate, mode-specific attribute (`relatedNonSingularBehavior`), when the correct approach is to recognize that generic fallback + hideIfEmpty logic handles all scenarios: all modes (automatic, manual, related) × all page types (singular, non-singular).

- **Type:** design-interpretation
- **Responsible Phase:** development

## Suggestions

### Project-Specific (applied to taktician.md)

1. **[ADD] Component Preferences**: Fallback and empty-state behavior in dynamic query blocks: Use a single generic `fallbackToLatest` attribute (not mode-specific) combined with `hideIfEmpty`. The fallback logic should work identically for both automatic and related modes, and should apply on both singular and non-singular pages. Do NOT create separate attributes like `relatedNonSingularBehavior` — instead, implement fallback in the query logic for all modes.
   - *Rationale:* Prevents attribute proliferation and ensures consistent behavior across all post selection modes and page contexts.
